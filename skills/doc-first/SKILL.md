# doc-first — 文档优先原则

## 1. 原则声明

> **文档优先原则（Docs-First Principle）**：任何关键信息必须先沉淀到文档，再开始编码；代码是实现文档意志的载体，而非唯一的真相来源。

本项目的 agent 在执行任何任务前，必须首先确认相关文档已存在；若不存在，应先协助创建或更新文档，再进入编码阶段。

## 2. 核心要求

### 2.1 先写文档，后写代码

- 新增功能前，必须在 `docs/architecture/overview.md`、`docs/adr/` 或 `docs/rfcs/` 中描述设计
- 新增接口前，必须在 `docs/api/` 中定义契约
- 新增命令或用户交互前，必须在 `docs/guide/` 或 `docs/COMMANDS.md` 中说明用法
- 不接受"代码即文档"作为借口

### 2.2 文档是项目交付物的一部分

- `README.md`、`AGENTS.md`、`docs/` 下所有文档与源代码同等重要
- 没有对应文档的代码变更视为未完成
- 代码审查时必须同时审查文档一致性

### 2.3 变更必须同步更新文档

| 变更类型 | 必须同步更新的文档 |
|---|---|
| 新增/修改接口 | `docs/api/`、`docs/guide/`、`README.md` |
| 新增/调整模块 | `docs/architecture/overview.md` |
| 引入新技术/依赖 | `docs/adr/`（新增 ADR） |
| 修改命令用法 | `docs/COMMANDS.md`、`docs/guide/` |
| 调整配置项 | `docs/CONFIGURATION.md` 或 `docs/ops/` |
| 变更部署方式 | `docs/ops/deployment.md` |
| 新增故障处理经验 | `docs/ops/troubleshooting.md` |
| 目录结构变化 | `AGENTS.md`、相关架构文档 |

### 2.4 文档面向两类读者

- **人类开发者**：快速理解、上手、维护
- **AI 协作者**：准确理解约束、减少幻觉

## 3. 文档优先级

当任务涉及多份文档时，按以下优先级处理：

1. **AGENTS.md**：项目级约定，任何变更前应先读取
2. **docs/introduction.md**：项目背景与核心概念
3. **docs/architecture/overview.md**：架构总览与模块职责
4. **docs/adr/**：关键架构决策记录
5. **docs/api/** 或接口文档：接口契约
6. **docs/guide/** 或 **docs/COMMANDS.md**：用户指南
7. **docs/ops/troubleshooting.md**：故障排查
8. **docs/changelog.md**：变更日志

## 4. 执行流程

agent 接到任务后，按以下流程执行：

```
1. 读取 AGENTS.md 和项目现有文档
   - 了解项目约定、技术栈、目录结构

2. 判断任务是否涉及文档变更
   - 若涉及新增/修改功能、接口、命令、配置、架构 → 必须同步处理文档

3. 确认或创建相关文档
   - 文档已存在且准确 → 基于文档继续
   - 文档缺失或过时 → 先创建/更新文档

4. 编写代码
   - 严格遵循已确认的文档约定

5. 验证文档一致性
   - 代码实现与文档描述是否一致
   - README / API 文档 / 指南是否已同步

6. 提交变更
   - 提交信息中若包含文档更新，使用 `docs:` 类型或同时标注
```

## 5. 常见任务处理规范

### 5.1 新增功能

1. 在 `docs/architecture/overview.md` 或 `docs/adr/` 中描述功能定位与接口
2. 在 `docs/guide/` 或 `docs/COMMANDS.md` 中补充用户侧说明
3. 创建/修改代码
4. 补充测试
5. 更新 `README.md`（如涉及安装/使用方式变化）
6. 更新 `docs/changelog.md`

### 5.2 修改接口

1. 先更新 `docs/api/` 中的接口契约
2. 再修改实现
3. 同步测试、示例、用户文档

### 5.3 重构或目录调整

1. 先更新 `AGENTS.md` 和 `docs/architecture/overview.md`
2. 再移动代码
3. 检查并更新所有引用该目录的文档

### 5.4 修复 Bug

1. 如 Bug 具有典型性，将复现步骤与解决方案写入 `docs/ops/troubleshooting.md`
2. 修复代码
3. 补充回归测试

### 5.5 技术选型决策

1. 将决策背景、候选方案、最终选择写入 `docs/adr/NNN-title.md`
2. 更新 `docs/architecture/overview.md` 中相关部分
3. 再引入依赖或实现

## 6. 检查清单

每次任务完成前，确认：

- [ ] 已读取 `AGENTS.md` 和现有相关文档
- [ ] 新增/修改的功能在文档中有对应描述
- [ ] 接口/命令用法文档已同步
- [ ] 架构文档或 ADR 已更新（如涉及架构变化）
- [ ] `README.md` 已更新（如涉及入口说明变化）
- [ ] `docs/changelog.md` 已更新（如适用）
- [ ] 代码实现与文档描述一致

## 7. 违规处理

当发现以下情况时，agent 应暂停编码并向用户说明：

- 用户要求直接写代码而跳过文档
- 现有文档与代码严重不一致
- 缺少关键架构文档或 ADR

建议话术：

> 根据本项目的文档优先原则，我需要先确认/更新 [具体文档] 后再编写代码，这样可以保证后续维护的一致性。是否允许我先处理文档？

## 8. 注意事项

- **不要替用户做无文档承诺**：任何未写入文档的约定都不应作为实现依据。
- **不要只更新代码不更新文档**：这是本 skill 禁止的核心行为。
- **保持文档最小可用**：文档不必长篇大论，但必须准确、可查找、可维护。
- **文档与代码同版本控制**：所有文档变更与代码变更一起提交。
